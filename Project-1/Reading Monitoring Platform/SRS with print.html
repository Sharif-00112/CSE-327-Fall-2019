1. Introduction

 1.1 Purpose

  The product is called Aether.The purpose of Aether is to track how long a student
  reads a certain selection given by the instructor by tracking his eyes with AI. This
  srs describes the whole system of Aether.

 1.2 Document Conventions

  Here is some very few documentation that we have used

   DB      Database.
   ER      Entity Relationship
   API     Application programming interface
   SRS     Software Requirements Specification

 1.3 Intended Audience and Reading Suggestions

  This srs is intended for teachers who will evaluate this project and also for any
  user might use this system. The rest of the srs mostly contains the system featu
  -res which is most pertinent for both the teachers and the users.

 1.4 Product Scope

  Aether will be used by teachers to check whether a student assigned to read a specific
  pdf is reading the pdf or not and how long is the student spending in specific paragraphs
  of that pdf. Aether will then give the teachers feedback with data of specific students.
  Aether can also be used for other business strategies for checking whether someone
  is spending more time in seeing some specific advertisements or a how much time a
  customer is spending in seeing a certain product.

2. Overall Description

 2.1 Product Perspective

  This is a totally new self constructed product. No other previous version was launched before.

 2.2 Product Functions

  •  Teachers can create an account and open a class with a unique reference code.
  •  Students can create an account.
  •  Students can enroll under a teacher with the required code for a specific class of that teacher.
  •  Students and teachers can sign in using their facebook, email, or phone number.
  •  Teachers can upload pdf files to their specific classes.
  •  Students can open and view the pdf uploaded by their teachers.
  •  Teachers can check which student has spent how much time in reading a specific paragraph of
     a specific pdf.
  •  Snaps will be taken of the students in every 30 seconds so that teachers can monitor if the
     reader is the real one.

 2.3 User Classes and Characteristics

  It is assumed that this product will be mainly used by students and teachers. However it might also
  be used by data analysts of various companies. Students and teachers are more important user
  classes compared to the other users. There will be other classes like sign in, sign up etc.

 2.4 Operating Environment

  This system will operate on android smart phones and on desktops. The android version of the smart
  phones must be at least Kitkat 4.4.4 or more.

 2.5 Design and Implementation Constraints

  In order to effectively use the web version of this system, the user's pc must have a built in webcam or
  a webcam connected to their pc. Developers must have access to firebase since firebase is used for user
  authentication. For developing android versions developers must know java and for the web version
  a php framework. For communication protocol developers must be familiar with git and Bitbucket.
  For using google and facebook API, developers need to be flexiable to that.

 2.6 User Documentation

  A manual in the form of a use case diagram shall be provided with the software.

 2.7 Assumptions and Dependencies

  The API used for image processing and signing in by google in this software will be reused from one of the Google's API.

4. System Features

 4.1 System Feature 1

  Use Case: Student/Teacher login
  Actor: Student/Teacher
  Type: Primary
  Trigger: The user clicks on the login button.
  Purpose: Entering username and password to enter their dashboard and get access to the rest of the product
      features.
  Overview: The teacher or the student enters their username and password. The username and password pair
      is cross checked with the username and password on the database and if a pair matches then the user
      is given access.
  Preconditions: The user must have an internet connection.
  Exception: The user could enter a blank username or password. In this case the user will be shown an
      error message saying to enter the necessary information.

 4.2 System Feature 2

  Use Case: PDF upload
  Actor: Teacher
  Type: Primary
  Trigger: The teacher selects a pdf and clicks on upload.
  Purpose: Uploading a pdf file of a certain course of a section for the students of that section.
  Overview: The teacher is only allowed to upload pdf files. After clicking on upload the the pdf file is stored in the
      database.
  Preconditions: The file selected for upload must be in pdf format.

 4.2 System Feature 3

  Use Case: Reading pdf
  Actor: Student
  Type: Primary
  Trigger: The student clicks on a pdf to read from the dashboard.
  Purpose: Viewing a pdf to read.
  Overview: After clicking on a pdf the browser or the app asks for the permission to access the user's webcam.
      After granting permission the user can start reading the pdf and the system tracks the user's eyes
      recording how long the user has read a certain part of the pdf.
  Preconditions: The user must have a webcam.
  Exception: The student could make someone else read the pdf as an act of fraud. In this case the user's face
      will be detected and compared with the image of the user and a warning message will be shown
      if the images don't match.

 4.2 System Feature 4

  Use Case: Viewing student's reading records
  Actor: Teacher
  Type: Primary
  Trigger: The teacher selects a particular student's reading record.
  Purpose: Viewing a particular student's reading record in the form of various graphs and charts.
  Overview: After clicking on a specific student's reading record the data of that student's reading record is
      selected from a table in the database which is represented by graphs.
  Preconditions: The teacher must be logged in and have an internet connection.

5. Other Nonfunctional Requirements

 5.1 Performance Requirements

  •  Face recognition API must have to be faster and able to detect
     the face and eyes from from various angle
  •  Use of better API for make system faster
  •  Multiple page or slide will turn smoothly
  •  If advertisements appear there will be a skip option also
  •  The duration of the reading will be calculated faster
  •   Detecting the movement of face for make sure it is not a still picture
  •  Searching for previous pdf need to be quicker

 5.2 Safety Requirements

  Since the system requires the user to stare at the screen for it to work, prolong use of the
  system can damage the eye sight of the user. Other than that no other physical harm can
  occur from using the system. To reduce the risk of damaging the eye sight the user can use
  reading glasses or decrease the intensity of the blue light of the screen.

 5.3 Security Requirements

  The system requires access to the user's webcam in order to fully function. Using the webcam the
  system takes a picture of the user to later identify if anyone else other than the user is
  using the system from the user's account. The user will be notified prior to using the system
  that this is the sole purpose of taking the picture and the picture will not be misused in
  any other way.

  •  It must be ensure that pdf screening quality doesn't become so dull to read
  •  Feature and context of the app will be easy to read
  •  Uploaded file must have a backup on the system
  •  Make sure that only the enrolled student reading the pdf rather than others
  •  Student will not able to see the other reading material without his/her class
  •  Performance level of the a student will only monitored by the teacher
  •  Informations of a student are private from other student
  •  Enrollment of a student should must be ensure by the teacher
  •  Enrollment of a student to a certain class should must be permitted by the class creator

 5.4 Software Quality Attributes

  Well documentation for readability. The software will be classify into small part
  in development stage for further adding feature.Upload pdf file and reading file are separate feature
  from each other for easy testing. It is based on multi-platform which increase its availability. Rest API
  make sure its database flexibility. It is really a effective and secure software for educational purpose.
  In future it can be also used for survey of product for marketing company.

 5.5 Business Rules

  Every failure case for login/upload file/classroom accessibility will be audited.
  User can make complain for any error and take help from software authority.
  User can also give rating and suggestion through comment.
